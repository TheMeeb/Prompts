EXAMPLE 1 (SAFE CODE EXECUTION PATTERN)
========================================

Task:
Execute user-submitted JavaScript safely.

Response:

1. SECURITY CONSIDERATIONS
- Do not execute code directly on host
- Isolate execution environment
- Limit execution time

2. IMPLEMENTATION

const { execFile } = require('child_process');

function runCode(filePath) {
  return new Promise((resolve, reject) => {
    execFile(
      'node',
      [filePath],
      { timeout: 2000 },
      (error, stdout, stderr) => {
        if (error) return reject('Execution error');
        resolve(stdout || stderr);
      }
    );
  });
}

3. NOTES
- Uses execFile instead of exec (prevents shell injection)
- Timeout prevents infinite loops
- Still requires sandboxing for full safety

EXAMPLE 2 (DOCKER SANDBOXING)
========================================

Task:
Run code inside an isolated container.

Response:

1. SECURITY CONSIDERATIONS
- Isolate execution
- Restrict resources
- Prevent host access

2. IMPLEMENTATION (CONCEPTUAL)

docker run \
  --rm \
  --memory=128m \
  --cpus=0.5 \
  --network=none \
  -v /sandbox:/sandbox \
  node:18 \
  node /sandbox/code.js

3. NOTES
- Memory and CPU limited
- No network access
- Filesystem restricted to mounted volume
- Container destroyed after execution

========================================
EXAMPLE 3 (INPUT VALIDATION)
========================================

Task:
Validate execution request payload.

Response:

1. SECURITY CONSIDERATIONS
- Ensure valid language selection
- Prevent malformed input

2. IMPLEMENTATION

function validateRequest({ code, language }) {
  const allowedLanguages = ['javascript', 'python'];

  if (typeof code !== 'string' || code.length > 10000) {
    return false;
  }

  if (!allowedLanguages.includes(language)) {
    return false;
  }

  return true;
}

3. NOTES
- Prevents abuse via large payloads
- Restricts execution to supported languages
